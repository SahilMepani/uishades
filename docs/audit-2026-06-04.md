## Headline resolution — the disputed login-CSRF finding is a FALSE POSITIVE

A prior review (`review-findings.json`, untracked) rated the magic-link POST
callback a **high** login-CSRF / session-fixation. Re-verification traced the
framework to ground truth and **overturned it**:

- `node_modules/@astrojs/cloudflare/dist/index.js:285` — `setAdapter(...)`
  unconditionally declares `adapterFeatures.buildOutput: "server"`.
- `node_modules/astro/dist/integrations/hooks.js:282-283` — propagates it:
  `settings.buildOutput = "server"` (overriding the `output:'static'` default).
- `node_modules/astro/dist/core/config/schemas/base.js:52` —
  `security.checkOrigin` defaults `true`; no override in `astro.config.mjs`.
- `node_modules/astro/dist/core/build/plugins/plugin-manifest.js:260` —
  `checkOrigin = (true && buildOutput === "server") = true`.
- `node_modules/astro/dist/core/base-pipeline.js:149-150` — registers
  `createOriginCheckMiddleware`.

So Astro's built-in origin check **is active in production**, and the app adds
an explicit stricter same-origin gate (`src/middleware.ts:106`). The callback is
double-protected and `loginUser` regenerates the session id. **No action; do not
re-raise.** The repo's own comments claiming "checkOrigin is a no-op under
`output:'static'`" are wrong — corrected in `CLAUDE.md`; the code comments
themselves (`csrf.ts:4-6`, `magic/callback.ts:16-17`, `csrf.spec.ts:2-3`) should
be fixed in a follow-up.

## Security

### Medium

- [ ] **Magic-link rate limiter is non-atomic (check-then-record TOCTOU)** —
  `src/pages/api/auth/magic.ts:66-77`, `src/lib/auth/db.ts:224-245`.
  `SELECT COUNT(*)` (L66-69) → gate (L70-72) → `INSERT` (L77) is split across two
  un-serialized statements with `await`s between them, and
  `magic_link_requests` has only a non-unique index. N parallel POSTs for one
  email/IP all observe `count < 5` and each sends an email, defeating the
  5/hr/email + /IP cap.
  *Impact:* inbox spam + faster burn of the shared Brevo ~300/day quota (soft DoS
  on magic-link sign-in). Bounded — tokens stay single-use (the consume path
  `db.ts:207-215` is correctly atomic via `DELETE…RETURNING`), so no auth bypass.
  *Fix:* record-then-check — INSERT first, then COUNT in-window and 429 (deleting
  the just-inserted rows) when over cap; or use a Cloudflare Rate Limiting
  binding / Durable Object.

### Low

- [ ] **Report flag-threshold tally silently capped to 1 hour** —
  `src/pages/api/palettes/[id]/report.ts:41-49,67`, `src/lib/auth/db.ts:243-245`.
  The tally declares all-time counting (`TALLY_SINCE = 0`), but every request
  first runs `pruneMagicRequests(db, now - 1h)`, whose `DELETE … WHERE
  created_at < ?` has **no key filter** and so wipes `report:<id>` rows older
  than 1h. Slow-accumulating legitimate reports never reach
  `REPORT_THRESHOLD = 3`. The L41 comment is false. (Same unscoped prune also
  runs from `magic.ts:61` and `feedback.ts`.)
  *Fix:* key-scope the prune so it never deletes `report:<id>`, or denormalize a
  `report_count` column; correct the comment.

- [ ] **`/api/palettes*` mutating routes absent from `CSRF_PROTECTED_PREFIXES`;
  `report.ts` docstring falsely claims upstream protection** —
  `src/lib/auth/csrf.ts:15` vs `src/middleware.ts:73`, `report.ts:18-19`.
  The app-level same-origin gate skips all palette mutations (create/PATCH/
  DELETE/vote/report). **No live exploit today** (SameSite=Lax strips cross-site
  cookies, create/PATCH require a JSON body, and Astro's now-confirmed-active
  built-in check blocks cross-origin form-like/bodyless POSTs incl. the bodyless
  vote) — but the documented invariant is false and untested, so a future change
  (form body, SameSite=None, a no-body route) would silently become exploitable.
  *Fix:* add `'/api/palettes'` to `CSRF_PROTECTED_PREFIXES` (`startsWith` already
  covers nested routes); add `csrf.spec.ts` cases for cross-origin POST to
  `/api/palettes` and `/api/palettes/<id>/vote`; fix the docstring.

- [ ] **Palette `description` bypasses the profanity guard `name` receives, then
  renders publicly/indexable** — `src/pages/api/palettes/[id].ts:65-68` vs
  `:57-63`. On PATCH, `name` runs through `isProfane` (L61); `description` is only
  length-capped. Palettes default `visibility='public'`; the description renders
  on `/p/[slug]` (body, `<meta name=description>`, OG/twitter, JSON-LD). Not XSS
  (Astro auto-escapes; JSON-LD via `safeJsonForScript`) — a content-moderation
  coverage gap on a Google-indexable surface. Bounded: page reachable only by the
  unguessable-suffix slug (`/explore` lists only `SEED_OWNER_ID`).
  *Fix:* mirror the name guard on the trimmed `description` before assigning
  `patch.description`.

## Performance

### Medium

- [ ] **Color-picker drag recomputes ramp+scale and re-renders the whole grid
  every `pointermove` frame** — `src/components/ShadeTool.tsx:480-481,494-512`,
  `ColorPicker.tsx:300-306`. `react-colorful` fires `onChange` per move;
  `handlePickerChange → handleChangeHex → setHex` has no throttle/rAF/
  `useDeferredValue`. Both `ramp` and `scale` `useMemo`s recompute every frame
  regardless of active view (~30 culori `toGamut` calls), no component is
  `React.memo`'d, and the sync effect does `history.replaceState` + a synchronous
  `localStorage.setItem` per frame. Jank on the tool's primary gesture on a
  mobile target.
  *Fix:* coalesce `onChange` to one rAF (or use `onChangeEnd`) or
  `useDeferredValue` the grid; `React.memo` on `ShadeRow`; debounce the URL/
  localStorage effect.

- [ ] **Image-mode point drag rebuilds every column's ramp/scale and re-renders
  ~160 swatches per frame** — `src/components/PaletteShadeGrid.tsx:70-76`,
  `ImagePalettePanel.tsx:259-281`, `ShadeTool.tsx:786-791,639-646`. Dragging a
  sample circle fires `onMovePoint` (no throttle) → new `tray` array identity →
  `paletteHexes`/`paletteNames` memos recompute → `columns` rebuilds
  `oklchRamp`/`buildScale` for **all** columns. `GridSwatch` is unmemoized and
  computes its label in render; `paletteNames` runs a 209-entry `nearestNamedSlug`
  OKLab scan per non-exact color. Worst case (OKLCH, 8 colors) ≈ 150 gamut maps +
  160 swatch re-renders/frame.
  *Fix:* throttle `onMovePoint` to one rAF; key the column memo by each column's
  own hex so only the dragged column recomputes; `React.memo` `GridSwatch`;
  compute `paletteNames` lazily / at export time.

- [ ] **N+1 D1 reads in `summarize()`** — `src/lib/auth/db.ts:948-955`. The
  per-row `await readPaletteColors(db, row.id)` loop is N sequential round-trips;
  the docstring's "avoids an N+1" claim covers only the batched votes query
  (L957-964). Up to ~101 sequential reads on the private/no-store dashboard and
  signed-in `/api/explore` (neither edge-cached).
  *Fix:* one `WHERE palette_id IN (…)` query grouped in JS (PK
  `(palette_id, position)` serves it), or `db.batch([…])` (already used at
  `db.ts:498/651/1013`).

### Low

- [ ] **`SignInModal` statically imported into the eager island chunk** —
  `src/components/ShadeTool.tsx:40`. Only renders behind `signInOpen` on a
  signed-out save; inconsistent with the file's lazy pattern. Realized saving
  ~1.5KB gz (the `SignInModal` wrapper; `AuthMenu` is already loaded by
  `HeaderAuth` on every tool route, so the ~8KB raw figure overstates it).
  *Fix:* `const SignInModal = lazy(() => import('./SignInModal'))` + `<Suspense>`.

- [ ] **Multi-color palette view computes each swatch's ramp/scale twice** —
  `ContinuousRamp.tsx:76-86` / `TailwindScale.tsx:71-81` vs
  `PaletteShadeGrid.tsx:70-76`. Export-groups memo and visible-grid memo both
  call `oklchRamp`/`buildScale` per hex with no shared cache; export memo runs
  even when the Export panel is closed. Bounded (≤8 swatches, sub-ms).
  *Fix:* lift per-swatch derivation into one parent memo, pass shades down.

- [ ] **Explore sort indexes don't cover `user_id`; trending sort is unindexed**
  — `src/lib/auth/db.ts:805-810,838-889`, `migrations/0004_palettes.sql:33-35`.
  Negligible at ~100-200 curated, edge-cached rows. Optional: add composite
  indexes leading with the constrained columns / materialize a trending score if
  the gallery grows.

## Completeness-critic gaps (surfaces the dimension scopes didn't clearly own)

1. **CSRF prefix-list seam** — covered above; the seam is that "middleware/CSRF"
   and the (previously undocumented) palettes feature each assumed the other
   owned the gate. Now documented in `CLAUDE.md`.
2. **Modal focus traps** — `SignInModal.tsx:97-100` and `HeaderAuth.tsx:211-214`
   declare `role="dialog" aria-modal="true"`, handle Escape/click-out/focus
   restore, but implement **no Tab/Shift+Tab containment** (WCAG 2.4.3 / APG
   dialog pattern). `tests/e2e/a11y.spec.ts` never opens a dialog, so axe can't
   catch it. Add a focus trap + an e2e case that opens the modal.
3. **`description` moderation gap** — covered in Security/Low above.
4. **`pruneMagicRequests` full-scan now on anonymous report/feedback hot paths** —
   `migrations/0002_rate_limit.sql:11` indexes only `(key, created_at)`; the
   never-pruned `report:<id>` rows grow the table that every report/feedback
   request runs an unscoped `DELETE … WHERE created_at < ?` over. Compounds the
   tally bug above; verify scan cost as report volume grows.
5. **`/explore` SSR page cache/`Vary` consistency** — `api/explore.ts:74-79`
   correctly branches `Cache-Control` on session + sets `Vary: Cookie`, but the
   SSR `/explore` page (`explore/index.astro`) renders the first page server-side
   with session-dependent `votedByMe` state. Verify it isn't edge-cached in a way
   that could serve one viewer's vote state to another (the page-level analog of
   the API caching).
